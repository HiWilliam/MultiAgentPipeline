# dual-cli-pipeline 手动跑通实施计划

日期: 2026-06-30
状态: 待执行
对应设计: [2026-06-30-dual-cli-pipeline-design.md](./2026-06-30-dual-cli-pipeline-design.md) §9 第 1 步

## 1. 目的

在写 `pipeline` 编排工具之前,先**纯手工**把两个 CLI(Claude Code / Codex)串起来跑一次,达成三件事:

1. 沉淀第一阶段文档模板(plan / review / fix 三类 artifact 的可填写骨架)
2. 验证三件事可操作:
   - 结构化 review 字段(`STATUS` / `ISSUES[]` / 收敛趋势)能被解析
   - 3a 自测门能真的拦住崩代码(测试红了不让进 3b)
   - 收敛趋势能手算(对比两轮 review 的 BLOCKER+MAJOR 数)
3. 暴露设计里没考虑到的实操摩擦,反哺设计文档

## 2. 试跑对象(刻意小)

选一个**自包含、可测、不超过两轮 fix 就能过**的小题目作为试跑内容,避免被业务复杂度干扰。

候选题目(选其一,执行时确定):
- A. 「写一个 python 函数 `slugify(s) -> str`,把任意字符串转成 URL 安全 slug,含测试」
- B. 「写一个 python 函数 `parse_kv(s) -> dict`,解析 `a=1,b=2` 形式字符串,含测试」

本题目的**故意埋坑点**(用于触发 3a 自测门和收敛监测):
- 在 plan 里写明一个边界用例(如「空字符串应返回空串」),让 Codex 第一版大概率漏掉 → 3a 自测门应红 → 自修。
- review 阶段再人为挑一个设计偏离点(如「未处理连续分隔符 `a--b`」),制造一轮 review FAIL → fix → 第二轮 review PASS。

## 3. 角色与工具

- **Claude Code(本会话)**: 充当阶段 1 设计 + 阶段 3b review。手写 plan、review 文档。
- **Codex CLI**: 充当阶段 2 实现 + 阶段 3a 自测 + 阶段 4 fix。由我手动调用。
- **人(我)**: 阶段切换、读文档、手算收敛趋势、记录摩擦。

> 注: 手动跑通阶段不写 `pipeline` 脚本,所有「驱动 / 解析 / 判定」由我手动完成,这正是为了发现脚本化时需要哪些输入。

## 4. 执行步骤

### 步骤 0: 准备目录
```
agent-pipeline/
└── runs/
    └── trial-001/                 # 本次试跑工作区
        ├── artifacts/
        │   ├── 001-plan.md
        │   ├── 002-review.md
        │   └── 003-fix.md
        ├── impl/                  # Codex 实现的工作目录
        └── state.manual.md       # 手动记录的状态(对应 state.json)
```

### 步骤 1: 阶段 1 设计(Claude Code,本会话)
- 我(本会话)按 §4.1 模板写 `001-plan.md`,含:
  - 有序实现步骤(2-3 步)
  - 每步: 目标 / 涉及文件 / **自测验收点(测试名或测试文件)**
  - 故意把边界用例写进验收点(埋坑)
- 产出: `artifacts/001-plan.md`
- 人工确认: 自己读一遍,判断「步骤是否可执行、验收点是否可机器判定」

### 步骤 2: 阶段 2 实现(Codex)
- 手动调 Codex,prompt: 「按 `001-plan.md` 实现,代码放 `impl/`,测试放 `impl/test_*.py`」
- 产出: `impl/` 下的代码 + 测试

### 步骤 3: 阶段 3a 自测门(Codex)
- 手动调 Codex(或直接 `pytest impl/`),跑测试。
- **验证点 1**: 若红(预期会红,因为埋了边界坑),让 Codex 读测试输出自修、再跑,直到绿。
- 记录: 自修了几次、每次红了什么。这是后续 `self_test_gate` 的输入数据。
- 绿了才进 3b。**若 3a 一直红 → 记录为摩擦点,人工介入。**

### 步骤 4: 阶段 3b review(Claude Code,本会话)
- 我(本会话)按 §4.2 模板写 `002-review.md`,含:
  - `STATUS: FAIL`(第一轮预期 FAIL)
  - 收敛趋势(首轮: 本轮 N / 上轮 N/A / 趋势 N/A)
  - 问题清单: 至少 1 个 BLOCKER(如「连续分隔符未处理」)+ 1 个 MAJOR
- 验证 `STATUS` 行格式可被正则解析。

### 步骤 5: 阶段 4 修复(Codex)
- 手动调 Codex,prompt: 「按 `002-review.md` 的 ISSUE 清单修 `impl/`,产出 `003-fix.md` 说明改了什么」
- 产出: 改后的 `impl/` + `artifacts/003-fix.md`
- 回阶段 3a(再跑测试)→ 3b(再 review)

### 步骤 6: 第二轮 3b review(Claude Code,本会话)
- 写 `004-review.md`,`STATUS: PASS`。
- **验证点 2**: 手算收敛趋势: 第一轮 BLOCKER+MAJOR=X,第二轮=0,趋势=下降。
- 记录: 手算过程是否顺畅、需要看哪些字段。

### 步骤 7: 收尾
- 把手动状态写进 `state.manual.md`(对应 §5.1 的字段)。
- 写「摩擦记录」(见 §5)。

## 5. 摩擦记录(必填)

试跑过程中,凡是「需要我动脑判断 / 字段不够用 / 模板缺项 / 流程卡住」的地方,都记下来,作为反哺设计的输入。最少回答:

1. **结构化 review 字段**: `STATUS` 行能否一眼解析?ISSUE 的 `类别` 枚举是否够用?有没有想写但模板没字段的问题类型?
2. **3a 自测门**: Codex 自修时,它需要哪些信息(测试输出全文?失败的用例名?)才肯改?自修几次后是否收敛?
3. **收敛趋势**: 手算时,只看 BLOCKER+MAJOR 够不够?MINOR 要不要计入?首轮无「上轮」怎么处理?
4. **artifact 流向**: fix 阶段读 review 时,Codex 是否真的逐条回应 ISSUE?有没有偷懒只改了一半?
5. **人工确认点**: 哪些确认其实是多余的(可自动化)?哪些本该确认但流程里没写?

## 6. 退出标准

试跑完成的标志:
- [ ] 一次完整跑通(设计→实现→3a→3b FAIL→fix→3a→3b PASS)
- [ ] 三类 artifact 模板(plan/review/fix)各有一份填好的实例
- [ ] `state.manual.md` 填完
- [ ] 摩擦记录 ≥ 5 条
- [ ] 据摩擦记录产出「设计文档修订项清单」(回写到 design 文档)

## 7. 时间盒

预计 1-2 小时。若超过 2 小时仍未跑通,停下来记录卡点,不要硬撑——卡点本身就是产出。

## 8. 试跑产出反哺设计的路径

试跑结束后,据摩擦记录更新 `2026-06-30-dual-cli-pipeline-design.md`:
- 模板缺项 → 补 §4.1/4.2/4.3
- 字段不够用 → 补 §5.1 state.json
- 流程卡住 → 调 §2 阶段图或 §5.2 升级条件
- 确认点多余/缺失 → 调 §7

更新后 design 文档状态从「设计待 review」→「设计 v2(经试跑校验)」,再进入 §9 第 2 步(实现 `pipeline` 工具)。
